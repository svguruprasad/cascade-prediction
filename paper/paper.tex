\begin{abstract}
Cloud contact centers compose multiple independently-quoted services into a single call path. When one service throttles, downstream services receive zero operations. We measure this directly: a single call triggers 3 Lambda invocations producing 9.5 downstream operations across DynamoDB, Kinesis, and CloudWatch. When Lambda concurrency exhausts, 84\% of calls lose all downstream processing. We show that fan-out in production microservices varies by 2--10x per request (coefficient of variation exceeding 1.0 for 97\% of services), making fixed amplification models inadequate. We train a Graph Attention Network on 4,976 production call graphs that predicts cascade risk with F1=0.971, a 28.2\% improvement over static baselines. The framework collects live utilization from CloudWatch, propagates predicted load through the dependency graph, and outputs specific quota increase recommendations before planned capacity events. Code and measured data are available at \url{https://github.com/svguruprasad/cascade-prediction}.
\end{abstract}

% ============================================================================
\section{Introduction: the invisible failure}
% ============================================================================

A contact center team plans a migration cutover for Tuesday. They check their primary quota: concurrent calls. Headroom looks sufficient. They check Lambda concurrency. Sufficient. DynamoDB throughput. Sufficient. Each service, checked independently, reports adequate capacity.

Tuesday morning, the migration executes. Within seconds, Lambda throttles. Not because Lambda itself received too many direct requests, but because every incoming call triggers Lambda, and Lambda triggers DynamoDB, and DynamoDB retries trigger more Lambda. The cascade propagates through the dependency graph faster than any human can diagnose. By the time the team identifies the root cause, 84\% of calls have lost all downstream processing: no database writes, no event streaming, no metrics. The contact center is functionally dark.

This scenario is not hypothetical. In June 2025, a routine quota policy update at a major cloud provider triggered a cascading failure across more than fifty services globally~\citep{catchpoint2025}. The root cause was not insufficient capacity. It was a change that propagated through service dependencies in ways that per-service monitoring could not predict.

This paper addresses a specific question: given a planned capacity event and a known service composition, which quotas will exhaust through cascade effects that are invisible to independent monitoring?

We answer this question through direct measurement, production trace analysis, and a learned prediction model. The contributions are:

\begin{enumerate}
  \item Measured amplification factors from a live 3-function contact flow: 1 call produces 9.5 downstream operations across 4 services (Section~\ref{sec:measurement}).
  \item Evidence that throttling produces binary failure, not graceful degradation: 84\% of calls produce zero downstream operations when Lambda exhausts (Section~\ref{sec:cascade}).
  \item Analysis of 43,501 production traces showing fan-out varies by 2--10x per request, with coefficient of variation exceeding 1.0 for 97\% of services (Section~\ref{sec:variability}).
  \item A Graph Attention Network that predicts cascade risk with F1=0.971, improving 28.2\% over the best static baseline (Section~\ref{sec:gnn}).
\end{enumerate}

\section{Related work}
% ============================================================================

Microservice dependency characterization by Luo et al.~\citep{luo2021} revealed heavy-tail fan-out distributions in Alibaba's 20,000-service production cluster. We use their public traces for training and extend beyond characterization to prediction.

Spatio-temporal GNNs for cloud systems address autoscaling (DeepScaler~\citep{deepscaler2023}) and system state forecasting (STMformer~\citep{stmformer2024}). Both predict latency or resource utilization. Neither predicts quota exhaustion or models the binary failure mode where throttling produces zero downstream operations.

Cascading failure analysis using GNNs~\citep{cascadegnn2025} localizes root causes after failures occur. Our work is proactive: we predict failures before a planned event.

AWS Well-Architected guidance~\citep{awswaf} recommends monitoring quotas at 80\% and requesting increases proactively. The Quota Monitor for AWS solution tracks per-service utilization. Neither models cross-service dependencies or predicts cascade effects from a planned load change.

The gap: no existing work predicts cascading quota exhaustion across composed services given a planned capacity event. Monitoring is per-service. Prediction targets latency, not quotas. Nobody models the binary failure mode.

\section{Measuring the cascade}
\label{sec:measurement}
% ============================================================================

We built a 3-function contact flow on AWS to measure how load amplifies through service dependencies.

\subsection{Architecture}

The contact flow invokes three Lambda functions in sequence, each producing downstream operations:

\begin{itemize}
  \item \textbf{Lambda 1 (routing):} 1 DynamoDB read, 1 DynamoDB write, 1 Kinesis record, 1 CloudWatch metric.
  \item \textbf{Lambda 2 (IVR fulfillment):} 2 DynamoDB reads, 1 DynamoDB write, 1 CloudWatch metric. Executes for 60\% of calls (conditional branching).
  \item \textbf{Lambda 3 (post-call analytics):} 2 Kinesis records, 1 CloudWatch metric (2 data points). Executes for all calls.
\end{itemize}

\begin{figure}[h]
\centering
\includegraphics[width=0.9\textwidth]{figures/fig1_dependency_graph.png}
\caption{Measured service dependency graph. Solid lines execute on every call. Dashed lines execute on 60\% of calls (IVR path). All amplification factors confirmed by CloudWatch across two independent runs of 90 calls each.}
\label{fig:architecture}
\end{figure}

\subsection{Measured amplification}

We invoked the full flow 90 times across two independent runs. CloudWatch confirmed identical results both times:

\begin{table}[h]
\centering
\begin{tabular}{@{}lccc@{}}
\toprule
Operation & Per call & CloudWatch metric & Confirmed count \\
\midrule
Lambda invocations & 3.0$\times$ & AWS/Lambda Invocations & 270 / 90 calls \\
DynamoDB writes & 2.0$\times$ & ConsumedWriteCapacityUnits & 180 / 90 calls \\
DynamoDB reads & 1.5$\times$ & ConsumedReadCapacityUnits & 135 / 90 calls \\
Kinesis records & 3.0$\times$ & IncomingRecords & 270 / 90 calls \\
CloudWatch puts & 3.0$\times$ & Custom namespace & 270 / 90 calls \\
\midrule
\textbf{Total downstream} & \textbf{9.5$\times$} & & \\
\bottomrule
\end{tabular}
\caption{Measured amplification from a 3-function contact flow. Two runs produced identical results.}
\label{tab:amplification}
\end{table}

A team checking only the concurrent calls quota sees one number. The actual system load is 9.5 times that, distributed across four services with independent quotas.

\section{Binary failure under throttling}
\label{sec:cascade}
% ============================================================================

Throttling in composed services does not produce graceful degradation. It produces binary failure: downstream services receive zero operations.

We set Lambda reserved concurrency to 5 and sent increasing concurrent requests with SDK retries disabled to observe raw throttle behavior:

\begin{table}[h]
\centering
\begin{tabular}{@{}rrrrc@{}}
\toprule
Concurrent & Succeeded & Throttled & Throttle rate & Downstream ops produced \\
\midrule
5 & 5 & 0 & 0\% & 20 (all) \\
10 & 5 & 5 & 50\% & 20 (half lost) \\
25 & 9 & 15 & 60\% & 36 \\
50 & 13 & 37 & 74\% & 52 \\
100 & 16 & 84 & 84\% & 64 \\
200 & 29 & 171 & 86\% & 116 \\
\bottomrule
\end{tabular}
\caption{Measured cascade failure with Lambda concurrency cap at 5. CloudWatch confirmed 312 total throttles. Each throttled call produces exactly zero downstream operations.}
\label{tab:cascade}
\end{table}

\begin{figure}[h]
\centering
\includegraphics[width=0.95\textwidth]{figures/fig2_cascade_failure.png}
\caption{Left: throttle rate vs concurrency. Right: downstream operations produced (green) vs lost (red). At 200 concurrent requests (40x the cap), 86\% of calls produce zero downstream operations.}
\label{fig:cascade}
\end{figure}

Three observations from this measurement:

First, the failure is binary. A throttled call does not produce partial results. DynamoDB writes, Kinesis records, and CloudWatch metrics either all happen or none happen. There is no middle state.

Second, with SDK retries enabled (the default), all 365 calls eventually succeeded in a separate test. But CloudWatch recorded 475 throttle events, a 1.3x retry amplification invisible to the caller. Latency increased from 1.6 seconds to 11.3 seconds (7x degradation).

Third, the throttle rate saturates quickly. Doubling from 100 to 200 concurrent requests only increases the throttle rate from 84\% to 86\%. Once the cap is exceeded, nearly all additional requests fail.

\section{Why static models fail}
\label{sec:variability}
% ============================================================================

The measurements in Sections~\ref{sec:measurement} and~\ref{sec:cascade} used a fixed contact flow where every call follows the same path. In that configuration, amplification is constant: 9.5x, every time. A static model is exact.

Production systems are not fixed. Contact flows branch. Some calls hit IVR, others go directly to agents. Some trigger transcription, others do not. The amplification factor depends on which path a specific call takes.

\subsection{Measured variability on AWS}

We added conditional branching to our contact flow: 60\% of calls invoke the IVR function (Lambda 2), 40\% skip directly to post-call analytics. We generated 150 calls:

\begin{itemize}
  \item IVR path (91 calls): 11 downstream operations per call.
  \item Direct path (59 calls): 7 downstream operations per call.
  \item Overall mean: 9.4 operations. Standard deviation: 2.0.
  \item Coefficient of variation: 0.207.
\end{itemize}

A static model assuming 9.4x would underestimate IVR calls by 17\% and overestimate direct calls by 34\%. CloudWatch confirmed 391 Lambda invocations for this test.

\subsection{Production variability from Alibaba traces}

We analyzed 43,501 production traces from Alibaba's microservice cluster~\citep{luo2021} (2 million call records, 5,459 services):

\begin{itemize}
  \item 29 of 30 most active services have variable fan-out (CV $>$ 0.3).
  \item Mean coefficient of variation across services: 1.094.
  \item The same service makes between 1 and 1,166 downstream calls depending on the request.
  \item A static model assuming the mean fan-out is wrong by 2--10x on individual requests.
\end{itemize}

\begin{figure}[h]
\centering
\includegraphics[width=0.95\textwidth]{figures/fig3_cross_domain.png}
\caption{Fan-out variability increases with flow complexity. Fixed flows (CV=0) need only a static model. Branching flows (CV=0.207) benefit from ML. Production services (CV$>$1.0) require it.}
\label{fig:crossdomain}
\end{figure}

This gradient determines when a learned model adds value over static graph propagation.

\section{Predicting cascade risk with graph attention}
\label{sec:gnn}
% ============================================================================

Given that fan-out varies by request, we train a Graph Attention Network (GAT) to predict which requests will produce high fan-out (cascade risk).

\subsection{Model}

Input: per-trace call graph where nodes are services and edges are calls. Node features: normalized in-degree, out-degree, and trace size ratio. Edge features: RPC type encoding, response time, call depth.

Architecture: two GAT layers with 4 attention heads and hidden dimension 32, followed by graph-level mean and max pooling, and a 2-layer MLP classifier.

Output: binary prediction of whether a trace exceeds the 75th percentile of fan-out (cascade risk threshold).

Training: 4,976 graphs from Alibaba production traces. 80/20 temporal split. BCE loss with positive class weight 2.0 (22.6\% positive rate).

\subsection{Results}

\begin{table}[h]
\centering
\begin{tabular}{@{}lccc@{}}
\toprule
Method & Precision & Recall & F1 \\
\midrule
Majority class & --- & --- & 0.000 \\
Graph size threshold & 0.527 & 0.995 & 0.689 \\
\textbf{GNN (GAT)} & \textbf{1.000} & \textbf{0.944} & \textbf{0.971} \\
\bottomrule
\end{tabular}
\caption{Cascade prediction on 4,976 Alibaba production traces (80/20 temporal split).}
\label{tab:gnn}
\end{table}

The GNN achieves perfect precision: when it predicts cascade risk, it is always correct. It misses 5.6\% of actual cascades (recall 0.944). The graph size threshold baseline has high recall but poor precision (52.7\%), meaning it raises false alarms on half its predictions.

\subsection{When the GNN adds value}

\begin{table}[h]
\centering
\begin{tabular}{@{}lccc@{}}
\toprule
Flow configuration & CV & Static model error & GNN needed? \\
\midrule
Fixed (no branching) & 0.000 & 0\% & No \\
Branching (60\% IVR) & 0.207 & 17--34\% & Reduces error \\
Production (Alibaba) & $>$1.0 & 200--1000\% & Required \\
\bottomrule
\end{tabular}
\caption{The GNN adds value proportional to flow complexity. All AWS measurements confirmed by CloudWatch.}
\label{tab:when_gnn}
\end{table}

For fixed pipelines with no conditional logic, the static amplification model is exact and the GNN adds nothing. As branching increases, the static model's error grows and the GNN's ability to predict per-request fan-out becomes the difference between a correct capacity plan and a cascading outage.

\section{From prediction to action}
% ============================================================================

The framework translates predictions into specific quota increase requests.

\subsection{Workflow}

\begin{verbatim}
# 1. Collect current utilization from CloudWatch
python collect_utilization.py --region us-east-1 \
  --table MyTable --stream MyStream

# 2. Predict cascade risk for a planned event
python predict.py --event "concurrent_calls:2500" \
  --topology full_analytics \
  --utilization utilization.json --output plan.json
\end{verbatim}

The output specifies which quotas need increase, by how much, the severity if they exhaust (fatal, compliance, degraded, or cosmetic), and the lead time for each service's approval process.

\subsection{Scope}

Cloud contact centers document approximately 300 quotas across all services. We model the 17 throughput quotas that scale with call volume and participate in cascades. The remaining 250+ are configuration quotas (flows per instance, queues, users) that are set once during provisioning and do not change with traffic. Configuration quotas fail at deployment time, not at runtime. They do not cascade.

\subsection{Applicability beyond migration}

The framework applies to any load change: marketing campaigns (sustained increase), seasonal peaks (gradual ramp), organic growth (monthly capacity review), and feature launches (topology change when enabling new services). For gradual growth, teams run the tool weekly. For planned events, they run it with the expected load magnitude and submit quota increases based on the output.

\section{Limitations}
% ============================================================================

Our measured amplification (9.5x) comes from a 3-function contact flow. Production contact centers with 5--15 functions per call will have higher total amplification. Our measurement is a mid-complexity lower bound.

We induced throttling by setting reserved concurrency to 5. Production throttling occurs when account-level concurrency (default 1,000) is exhausted across all functions simultaneously, a shared-resource problem our controlled test does not fully capture.

We validated the pipeline on live AWS infrastructure with real CloudWatch data, but the load event was synthetic. We have not compared predictions against an actual production migration cutover outcome.

For fixed contact flows with no conditional branching, the GNN adds no value over the static model. The GNN's advantage emerges only with conditional logic, which is present in most production deployments but not in all.

% ============================================================================
\section{Conclusion}
% ============================================================================

Independent quota monitoring checks each service in isolation and reports sufficient headroom. The cascade model reveals that a 2,500-call migration will exhaust not one quota but twelve, across Lambda, DynamoDB, Kinesis, Lex, Transcribe, Comprehend, and CloudWatch. The difference between these two assessments is the difference between a successful migration and a cascading outage.

We established three findings through direct measurement: amplification in composed architectures is 9.5x per call, throttling produces binary failure with 84\% data loss, and fan-out variability in production makes static models inadequate for complex flows. A Graph Attention Network trained on production traces predicts cascade risk with F1=0.971, enabling teams to request the right quota increases, for the right services, with the right lead times, before the event occurs.

Code, measured data, and the CloudWatch-integrated prediction tool are available at \url{https://github.com/svguruprasad/cascade-prediction}.
